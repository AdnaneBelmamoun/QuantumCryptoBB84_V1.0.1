Reconciled B92 Quantum Transmission

(initiator's side)


A reconciled B92 key is obtained after applying a reconciliation protocol to a plain B92 key. Reconciliation is an interactive process taking place over the classical channel and aiming at correcting the errors between the sender and the receiver keys while disclosing as less bits as possible. Reconciliation is performed by an interactive protocol called Cascade. You can find more about reconciliation and the Cascade protocol here.

Plain Key Generation

The first step in order to get a reconciled key is to first generate a plain key which is subject to transmission errors and only partially private:
  • PlainB92

The PlainB92 subprotocol allowed Alice and Bob to define a plain key of 276 bits. The next step aims at correcting the errors between the sender's and receiver's version of the plain key.

Error-Correction Using Cascade


Cascade is an interactive and parity-based error-correction scheme. Before running Cascade, the inial blocksize for parity testing must be chosen together with the number of passes.

The initial blocksize k0 is chosen to be k0=(1/e)+(1/4e) where e is the error estimate provided by the plain key generation. The blocksize ki+1 for pass i+1 is defined as ki+1=2ki. The last normal pass occurs when the blocksizes exceeds 1/4th of all bits.Two extra passes are used with blocksize about n/4. Note that blocksize never exceeds 1/4th of all bits.

Given an an error-estimate of 0.093333334 and a B92 plain key of size 276 , we get:

  • the initial blocksize k0= 11 ,
  • the number of passes is 7 .
Cascade fixed a total of 23 errors revealing 148 parity bits, where:
  • 11 errors were corrected in pass 0 using block of size 11 revealing 60 parity bits.
  • 12 errors were corrected in pass 1 using block of size 22 revealing 65 parity bits.
  • 0 errors were corrected in pass 2 using block of size 44 revealing 7 parity bits.
  • 0 errors were corrected in pass 3 using block of size 88 revealing 4 parity bits.
  • 0 errors were corrected in pass 4 using block of size 88 revealing 4 parity bits.
  • 0 errors were corrected in pass 5 using block of size 88 revealing 4 parity bits.
  • 0 errors were corrected in pass 6 using block of size 88 revealing 4 parity bits.

The next phase is the confirmation of the equality of the initiator's and responder's reconciled keys. If Cascade has succeeded then the next step should confirm it.

Confirmation

In order to make sure that no error remains, the parity of random subsets of positions are exchanged and compared. In general, if w parity bits are compared and found identical then the initiator's and the responder's keys are identical except with probability 2-w. In this experiment,
  • Confirmation by publicly comparing 10 parity bits was performed.
  • Confirmation is successfull.
  • Alice's and Bob's keys are identical except with probability 2-10,which is about 9.765625E-4.

Statistics & Overview

Here are some statistics about this execution.
  • The observed error-rate is 0.083333336 compared to the expected value 0.093333334 produced by error-sampling.
  • The total number of parity bits disclosed during both reconciliation and confirmation is 158 .
  • The reconciliation rate is R=(n-c)/n where c is the total number of parity bits revealed and n is the size of the plain B92 key. For this execution we have R= 0.42753622 compared to the optimal Shannon bound Rmin= 0.58618313 .
  • Reconciliation has produced identical keys with probability 0.99902344 .
